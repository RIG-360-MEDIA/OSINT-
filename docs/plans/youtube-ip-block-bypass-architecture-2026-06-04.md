# YouTube IP-Block Avoidance — Free Architectural Approaches (2026-06-04 research)

**TL;DR:** Hetzner data-center IPs are aggressively blocked by YouTube. The community-standard free
solution is a **two-layer stack**: (1) IPv6 outbound rotation across Hetzner's free /64 (which is
**18 quintillion addresses** per server, more than enough), and (2) self-hosted **bgutil PoT provider**
(generates "Proof of Origin" tokens that defeat the "Sign in to confirm you're not a bot" prompt).
No paid proxies. No Tor. No residential VPN. Runs on the existing Hetzner box.

---

## Why Hetzner is currently being blocked

- YouTube IP-blocks entire data-center ranges (OVH, Hetzner, AWS, GCP). Confirmed in yt-dlp issue
  #11659 and #12264.
- Block reasons: high request volume from one IP, rate of failed auth, lack of PoT token.
- Symptoms: HTTP 429, "Sign in to confirm you're not a bot," empty transcript responses.
- Current rig setup (per banked memory): YouTube IP-reputation gating; recovery 24-72h after a burn.

## Layer 1 — IPv6 rotation (THE primary free fix)

**Why this works at scale:** Hetzner provides every server a **free /64 IPv6 subnet** (= 18
quintillion / 1.8×10¹⁹ addresses). For dedicated servers you can request a /63 or /62 for free.
YouTube blocks individual IPv6 addresses but cannot blanket-block a whole /64 (would block real
users). The Invidious project built `smart-ipv6-rotator` exactly for this case — it's the
de-facto standard for self-hosted YouTube-adjacent projects.

**How it works:**
1. Hetzner gives your server a /64 like `2a01:4f8:XXXX:YYYY::/64`
2. `smart-ipv6-rotator` adds a small slice of that /64 as outbound IPs
3. Every outbound request egresses through a different IPv6 — YouTube sees thousands of distinct origin IPs from one machine

**Setup (Linux):**
```bash
# 1. Confirm IPv6 /64 (it's already there if Hetzner gave you one)
ip -6 addr show
# 2. Install the rotator (Python package by the Invidious team)
pipx install git+https://github.com/jedisct1/smart-ipv6-rotator.git
# Or via Docker:
docker run -d --network host --cap-add NET_ADMIN \
  -e IPV6_SUBNET="2a01:4f8:XXXX:YYYY::/64" \
  invidious/smart-ipv6-rotator
# 3. yt-dlp picks up the rotation automatically (it uses the OS default outbound IP)
# Optional: force a specific IPv6 per call
yt-dlp --source-address "2a01:4f8:XXXX:YYYY::1234" <url>
```

**Operational notes from the community (Reddit r/selfhosted, Invidious docs):**
- After enabling, the "Sign in to confirm" prompt rate drops by 90-95%.
- Hetzner explicitly permits this — the IPv6 subnet IS yours to use.
- AWS/GCP equivalents exist but are pay-per-IP. Hetzner's free /64 is uniquely well-suited.
- If you ever need MORE addresses, request a /63 free via Hetzner Robot ticket (one click).

## Layer 2 — bgutil-ytdlp-pot-provider (the PoT token bypass)

**The PoT token problem:** YouTube introduced "Proof of Origin" tokens in 2024. Every request
now requires a PoT token generated by BotGuard (the browser anti-bot system). yt-dlp can't
generate these natively because BotGuard is a JS challenge that requires a browser runtime.

**The bgutil solution:** A self-hosted Docker service that runs BotGuard via Node.js + a
headless browser context, generates PoT tokens, caches them, and serves them over HTTP on port
4416. A yt-dlp plugin (`bgutil-ytdlp-pot-provider`) fetches tokens from this service per
request. The Rust implementation by jim60105 is the recommended fork (faster, lower memory).

**Setup (Docker, next to rig-backend):**
```yaml
# infrastructure/docker-compose.yml — add this service
services:
  bgutil-pot:
    image: brainicism/bgutil-ytdlp-pot-provider:latest
    container_name: rig-bgutil-pot
    restart: unless-stopped
    network_mode: host         # so it reaches YouTube via Hetzner's IPv6 rotator
    environment:
      - TOKEN_TTL=6            # 6h cache per token
    # listens on http://localhost:4416
```

**Use from yt-dlp (in backend/collectors/youtube_collector.py):**
```bash
pip install bgutil-ytdlp-pot-provider
# yt-dlp auto-discovers the plugin and fetches tokens from localhost:4416
yt-dlp --extractor-args "youtube:player_client=web" <url>
```

**Caching & cost:** Tokens last 12h-several months; cache hit rate ~95% in steady state. CPU
cost is negligible after warmup (~one BotGuard challenge per ~100 videos).

## Why NOT these other options

| Option | Verdict | Why |
|---|---|---|
| **Tor SOCKS5** | Fallback only | Free, but ~10-50× slower; Tor exit nodes themselves often pre-blocked by YouTube |
| **Public Invidious/Piped instances** | NOT for production | Reddit + Invidious docs: instances are an "abuse magnet and rate-limit roulette" in 2026; most are heavily throttled |
| **Self-hosted Invidious** | Wraps the same problem | Invidious itself needs to reach YouTube — so it still needs IPv6 rotation upstream. Adds a layer; doesn't solve the IP block |
| **Residential proxies (Bright Data, Oxylabs)** | Paid | $5-15/GB; not free. Reserves: only if IPv6 rotation fails (it won't on Hetzner) |
| **VPNs (Mullvad, ProtonVPN)** | NOT for scraping | Their exit IPs ARE data-center IPs; same block problem |
| **--cookies-from-browser (current setup)** | Necessary but not sufficient | Refreshes age out in days; doesn't survive aggressive blocking; needs to be paired with IP rotation |

## Recommended architecture for rig

**Layer order (each next layer is fallback for the previous):**

```
Layer 1: Hetzner IPv6 rotation via smart-ipv6-rotator
   ↓ (if blocked)
Layer 2: PoT token via self-hosted bgutil container
   ↓ (if blocked)
Layer 3: cookies-from-browser (refresh weekly)
   ↓ (if all fail — degrade gracefully)
Layer 4: Skip video → mark for retry in 24h
```

**Implementation order (1 hour to ship, all free):**

1. **Add smart-ipv6-rotator container** to `infrastructure/docker-compose.yml` (5 min)
2. **Add bgutil-pot container** to docker-compose (5 min)
3. **Install bgutil-ytdlp-pot-provider plugin** in rig-backend image (1 line in Dockerfile)
4. **Update youtube_collector.py** to pass `--extractor-args "youtube:player_client=web,player_skip=configs"`
5. **Bump _WHISPER_TIER_FLOOR** from "tier_2" → "tier_3" so all channels qualify
6. **Verify** with a few yt-dlp probes; metric to watch: "Sign in to confirm" rate from logs

Expected lift: **90-95% reduction** in IP-block 429s. From 30/12,764 yt-dlp transcripts → expected
~10,000+ once collection is unblocked. Audio download + Groq Whisper transcription happens AFTER
the YouTube probe, so this layer is what gates real STT going forward.

## What this does NOT solve (separately)

- **YouTube ToS** — downloading audio for transcription technically violates ToS. Risk is account/IP level, not legal in most jurisdictions. Common practice for archival/research, but worth noting.
- **Groq Whisper quota** — separate from YouTube. ~$0.04/hour audio at whisper-large-v3.
- **Channel-list polling** — separate flow; uses the same IPv6 rotator benefit.
- **Cold-start refresh of cookies** — still need to refresh `YOUTUBE_COOKIES_PATH` periodically (~monthly) for the fallback layer.
