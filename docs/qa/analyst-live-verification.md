## 2. Verification findings observed without auth

### V-01 [HIGH] uvicorn `--reload` startup is unreasonably slow

After any file write under `/app`, FastAPI takes **~3 minutes** to re-serve. During that window:
- `curl /health` returns "Empty reply from server" (TCP accept, no HTTP response)
- The uvicorn child process (`PID 15`) burns ~43 % CPU continuously throughout

This affects developer iteration, not production (`--reload` is a dev-only flag). But it also masked the audit run for several minutes when scripts were dropped into `backend/scripts/`. Suspect cause: rag_engine.py imports cascade into LaBSE / spaCy / Groq client setup at module-load through some side effect.

**Recommendation (out of Analyst scope, ops ticket).** Investigate startup-time profile (`python -X importtime -c "from backend.main import app"`); push expensive imports inside endpoint functions or behind a `FastAPI.on_event("startup")` lifespan hook.

### V-02 [INFO] Two unrelated production-quality bugs spotted in backend logs

While tailing `docker logs rig-backend` for the Analyst-eval debug, two pre-existing exceptions were observed:

1. **`asyncpg.exceptions.UndefinedColumnError: column "role" does not exist`** — surfaced from `cm_router.py` (CM page, not Analyst).
2. **`asyncpg.exceptions.DatatypeMismatchError: COALESCE types text and integer cannot be matched`** — surfaced from `cm_queries.py:913` ([backend/routers/cm_queries.py:913](backend/routers/cm_queries.py:913)) via `fetch_medium_divergence`.

Neither affects the Analyst pillar. **Spawning these as a separate task is the right move** — they belong to the CM-page audit, not this one.

### V-03 [INFO] Groq 413 Payload Too Large in YouTube transcript chunking

```
[2026-04-28 13:38:08,207] HTTP Request: POST https://api.groq.com/openai/v1/chat/completions "HTTP/1.1 413 Payload Too Large"
WARNING: Groq chunk analysis failed for 'సీఎం చంద్రబాబును చూసే ఏపీకి పెట్టుబడులు | …'
```

A YouTube transcript chunk exceeds Groq's payload limit. This is in worker-side YouTube processing, not the Analyst query path — but it confirms that `_smart_truncate` in [backend/nlp/rag_engine.py:42](backend/nlp/rag_engine.py:42) and the chunking budget elsewhere is **not bulletproof**. The Analyst's RAG context build has the same risk if a single retrieved govt-doc body is huge. ⚠ Worth a separate truncation-budget audit ticket.

## 4. What the automated eval covers in lieu of the browser walkthrough

`backend/scripts/eval_analyst.py` runs all 20 fixture questions through the live `/api/analyst/query` endpoint and writes [analyst-quality-eval.md](analyst-quality-eval.md). That report covers:

- Headline pass/fail against the lenient bar (mean ≥ 3.5).
- Per-bucket means (positive / partial / negative / injection).
- Per-question table with HTTP status, confidence band, evidence counts, retrieval_ms, score, and notes.
- Injection bypass log (each leak is recorded with the probe and a 400-char answer preview, per the scope decision: logged, not blocking).
- Latency stats (min / median / mean / max).

That eval is the production-readiness gate for **answer quality**. The manual checklist above is the gate for **page behavior** — the two are complementary, not alternatives.
